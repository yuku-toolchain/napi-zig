# Manual setup

If you would rather wire up a project by hand, or you are adding napi-zig to a repo that already exists, this page is the complete path. It is a 1:1 of what `napi new` writes, plus pointers into the publishing flow when you are ready to ship.

## 1. Add napi-zig

```sh
zig fetch --save git+https://github.com/yuku-toolchain/napi-zig.git/#HEAD
npm install -D napi-zig
```

The first command pins napi-zig as a Zig dependency in `build.zig.zon`. The second installs the `napi` CLI as a dev dependency. Use `pnpm`, `yarn`, or `bun` if you prefer; the CLI is the same.

## 2. Write your addon

```zig
// src/lib.zig
const std = @import("std");
const napi = @import("napi-zig");

comptime { napi.module(@This()); }

pub fn add(a: i32, b: i32) i32 {
    return a + b;
}

pub fn greet(env: napi.Env, name: []const u8) ![]const u8 {
    return std.fmt.allocPrint(env.allocator(), "Hello, {s}!", .{name});
}
```

`napi.module(@This())` walks every public declaration of the file at comptime:

| Declaration                           | Becomes               |
| ------------------------------------- | --------------------- |
| `pub fn name(...)`                    | A JS function         |
| `pub const x = <JS-mappable value>`   | A JS property         |
| `pub const x = struct { pub fn ... }` | A nested JS namespace |

Names are translated `snake_case` to `camelCase` automatically. See [Functions](/functions) for the full rules.

## 3. Configure `build.zig`

```zig
const std = @import("std");
const napi_zig = @import("napi_zig");

pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    napi_zig.addLib(b, b.dependency("napi_zig", .{}), .{
        .name = "my-addon",
        .root = b.path("src/lib.zig"),
        .target = target,
        .optimize = optimize,
        .npm = .{
            .scope = "@myscope",
        },
    });
}
```

`addLib` is the only thing you need to call. It registers an artifact, sets up the right linker flags for each OS, and (when `--release` is set) generates the cross-compile graph and the npm package skeleton. See [build.zig (addLib)](/reference/build) for every option.

The `.scope` you pick here is the npm scope your per-platform bindings publish under. Set it to a username or org you own, and create the org on npm before publishing. See [Publishing: how distribution works](/publishing#how-distribution-works) for the full rationale.

## 4. Add scripts to `package.json`

```json
{
  "name": "my-addon",
  "version": "0.0.0",
  "type": "module",
  "scripts": {
    "build": "napi build",
    "release": "napi build --release",
    "bump": "napi bump",
    "test": "node test.mjs"
  },
  "devDependencies": {
    "napi-zig": "^0.1.0"
  }
}
```

The version field starts at `0.0.0`. `napi bump` manages it from here on.

## 5. Add a `.gitignore`

```
node_modules
zig-out
.zig-cache
zig-pkg
.DS_Store
*.tgz

# regenerated by `napi build`
/my-addon.js
/my-addon.d.ts
```

Replace `my-addon` with your addon's `.name`. The two project-root files are re-exporters that `napi build` writes so `import addon from "./my-addon.js"` works locally; they are derived artifacts and should not be committed. The `npm/` directory **is** committed once you start publishing; it is the publishable tree, kept in sync with `build.zig` by every release build.

## 6. Build and use

```sh
napi build
```

```js
// test.mjs
import addon from "./my-addon.js";

console.log(addon.add(2, 3)); // 5
console.log(addon.greet("world")); // "Hello, world!"
```

```sh
node test.mjs
```

That is the full development loop. Edit `src/lib.zig`, re-run `napi build`, re-run your test.

## 7. Set up publishing (when you are ready)

When you want to ship the addon to npm, you need two more pieces in place:

- A **CI workflow** that publishes per-platform binaries on tag push. Save the YAML from [Publishing: the CI workflow](/publishing#the-ci-workflow) as `.github/workflows/publish.yml`.
- The **publish flow itself**: cross-compile, push to GitHub, run `napi npm-init` once, then `napi bump` for every release.

Walk through [Publishing to npm](/publishing) end to end. It assumes you reached it from this page or from `napi new`; the steps are the same either way.
